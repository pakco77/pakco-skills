# Agent Team 测试方法论

## 三阶段测试策略

对每种需求阶段（0→1 / 1→2 / 小需求）各走一遍完整链路，验证 Agent Team 设计的逻辑闭环。

### 测试方式

1. **扮演 PMO**：做 Socratic 对话，产出派发描述
2. **按链路逐 Agent 产出**：写入对应 feature 文件夹的 B 级文档
3. **FE 必须产出真实 HTML 原型**：否则 FQA/VQA 无法测试
4. **FQA + VQA 并行审查**：各自独立产出报告
5. **验证微闭环**：若有 FAIL，验证闭环归属判定是否正确

### 关键验证点

- DA 阶段判定是否准确分叉 pipeline
- HW(P1) 是否正确识别硬件关联（不涉硬件时一行通过）
- HW(P2) 是否基于 S 级真实硬件参数产出红绿灯
- UX 是否为每个 HW 黄灯给出交互限制映射
- UI 的 CSS 变量是否与 FE 形成强契约
- FE 的「验收标准映射」是否让 FQA 可逐条复现
- FQA 和 VQA 的报告是否零重叠（分工正确）
- 问题归属判定是否正确（表层 vs 结构级）

### 发现的典型问题

| 问题 | 类型 | 发现的测试 |
|------|------|-----------|
| 重复 DOM id 导致 JS 更新遗漏 | 表层 FE bug | 测试 2 (1→2) |
| 硬编码色值未用 CSS 变量 | 表层 FE 偏离 | 测试 1 (0→1) |
| 刀路颜色/箭头/缩放未实现 | 验收标准未通过 | 测试 1 (0→1) |
| 组件缺失 Loading 状态 | 规范偏离 | 测试 1 (0→1) |
| 错误路径不可达 | 边界缺失 | 测试 1 (0→1) |

### 设计缺口（已确认，待修）

1. PMO 缺「读 DA 阶段判定后分叉 pipeline」的显式逻辑（小需求应跳 HW(P2)）→ **已修（2026-05-09）**
2. 跨 feature STALE 通知机制缺失

## 完整 0→1 实战案例：Genmitsu Web CAM MVP（2026-05-09）

最近一次完整 0→1 执行，产出可作为"期望质量"参考：

| 产物 | 指标 |
|------|------|
| Roadmap | 8 个 P0 + 5 系统级决策 + 竞品对标 |
| PRD | 8 个 P0 展开为 33 条验收标准 |
| UX | 6 页面（首页/Wizard×4/画布）+ 状态定义 + HW 限制映射 |
| UI | CSS 变量系统 + 首页/Wizard/工作台专用变量 |
| FE 原型 | 28KB 单文件，含拖拽/缩放/文字编辑/刀路生成/导出全交互 |
| 测试 | FQA 28/33 通过，VQA 5/7 维 PASS |

**执行要点**：用户说"启动"即全链路跑通，3 个用户介入点（Roadmap确认/黄灯裁决/原型验收）均暂停等待。黄灯裁决时用户可推翻默认材料选择——PMO 应逐条呈现并接受变更。
