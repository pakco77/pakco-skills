# Prototype-First Feature Iteration for Genmitsu CAM

## 触发信号
当用户明确表达以下任一口径时，进入“原型优先”节奏，而不是把文档冻结当成唯一终点：
- “以原型做出来为最终目的”
- “可以分批执行来迭代原型”
- “先把某个 feature 做进原型，再处理下一个模块”

## 核心原则
1. 文档是约束输入，不是交付终点。
2. 允许按 feature 小闭环推进：一个 feature 在原型里落地并可验证后，再进入下一个 feature。
3. 进度汇报必须区分“文档状态”和“原型状态”，避免只说文档已改却没有原型落地。

## 推荐节奏
1. 锁定当前 feature 与目标原型文件。
   - 如果会话已建立“新文件不覆盖旧原型”的策略，继续沿用该副本文件。
2. 只对当前 feature 做最小必要的上游文档修订。
   - 需要区分新旧时，先新建 sibling 副本（如 `_GPT5.4xh`）再改。
   - 不要求把所有受影响 feature 文档一次性补齐后才进入 FE。
3. 立即把当前 feature 落进原型 HTML。
4. 对该 feature 做局部验证。
   - 至少检查：入口文案、路由/跳转、状态切换、关键反馈、视觉层级。
5. 记录当前 feature 已落地什么、未落地什么，再切到下一个 feature。

## 对 F000 / F001 这类耦合模块的用法
- 如果 F000 依赖 F001 容器或路由槽位，允许先在原型里把 F001 必要承托层做出来，再回到 F000 首页本体。
- 判断标准不是“文档顺序必须绝对线性”，而是“当前原型能否继续有效收敛”。

## 进度汇报模板
- 已完成：哪些文档副本已落盘，哪些原型能力已在 HTML 中出现
- 进行中：当前正在改的 feature
- 未完成：还缺哪些文档修订 / 原型落地 / 验证
- 下一步：直接指向下一个会动手的文件

## 常见误区
- 只补 PRD/UX/UI，不推进原型 HTML
- 只说“文档已对齐”，不说明原型是否已经体现
- 因为多个 feature 互相影响，就无限延后 FE 落地
- 把“文档全量修齐”误当成进入原型的硬前置
