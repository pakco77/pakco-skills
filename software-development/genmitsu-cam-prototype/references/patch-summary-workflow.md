# Patch Summary Workflow for Genmitsu CAM Agent Team

## 何时触发
当 PMO 苏格拉底阶段已经锁定一批会同时影响多个冻结文档的裁决，而且下游正式文档仍沿用旧口径时，先进入“补丁方案整理”阶段，而不是立刻逐份重写文档。

典型信号：
- 同一批裁决同时影响 L1 / F000 / F011 这类多份文档
- `_product/log.md` 已连续记录多条用户裁决
- 当前 feature 文档仍写着旧 CTA、旧路由、旧锚点或旧叙事
- 某个下游 feature（如 F011）尚未建档，但已被新裁决明显影响

## 标准动作
1. 先读 `_product/log.md`，确认最新已锁裁决与影响范围。
2. 再读 L1 与相关 feature 的当前冻结文档，找出仍沿用旧口径的位置。
3. 输出 `_product/YYYY-MM-DD-...-patch-summary.md`：
   - 先写“已锁裁决”
   - 再写“现状扫描结论”
   - 再写“对各文档的补丁要求”
   - 最后给“推荐派发顺序”
4. 立即把这次摘要落盘，并在 `_product/log.md` 追加一条 `生成补丁摘要` 记录。
5. 按 `L1 → F000 PRD → F000 UX/UI → F011` 的顺序派发正式修订。

## 为什么这样做
- 先统一口径，避免下游继续沿用旧的 CTA / 路由 / 锚点叙事。
- 让 F011 这类尚未建档的 feature 也有稳定输入基线。
- 减少“边改边发现前提不一致”导致的返工。

## 版本副本与日志纪律
当用户要求“区分新旧”时：
1. 不覆盖旧冻结文档。
2. 在同目录新建 sibling 副本，并加明确后缀（如 `_GPT5.4xh`）。
3. 后续正式修订都基于这个新副本继续，而不是回头覆盖旧文件。
4. 在 `_product/log.md` 追加一条“新建副本 / 保留旧版”的记录，写清新文件路径。

## 写 `_product/log.md` 的工具注意事项
- 先重读整份 `log.md`，不要只依赖 offset/limit 的分页片段。
- 如果工具提示 sibling subagent / partial-read warning，先完整读文件再写，避免误覆盖其他并行改动。
- 记录副本类动作时，推荐同时写明：旧版保留、当前后续修订目标、新文件名后缀来源。

## 这次会话新增验证点
- 当用户明确要求“新建文档，区分新旧”时，这不是单次命名偏好，而是冻结产物修订策略：后续所有受影响文档都应沿用“保留旧版 + 新建带后缀副本”的方式继续。
- `_GPT5.4xh` 这类后缀可作为一次会话/模型来源标识，帮助 PMO 在多轮接管中快速辨认版本来源。

## 这次会话验证出的关键点
- Home 升级为 Step 0 时，F000 不再只是“项目入口页”，而是“新手首雕入口页”。
- F011 不再只等于 workspace 内 spotlight，而要前移承接首雕 Step 1 / Step 2。
- 如果当前冻结产出中未发现 F011 文档，补丁摘要必须显式标注“未建档”，并把自己作为首次建档输入基线。
